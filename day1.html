Deep Dive: The Agent Loop

How does an agent actually "think"? It is a continuous loop of Observation, Thought, and Action.

The ReAct Pattern

Most modern agents follow the ReAct (Reason + Act) architecture:

Input: The agent receives a high-level goal.
Thought: The LLM generates an internal monologue explaining what to do next.
Action: The agent generates a structured command (JSON) to call a tool.
Observation: The tool executes (e.g., Python runs) and returns output to the agent.

Under the Hood

while (goal_not_met):
  
  # 1. Ask Brain
  response = LLM(context + history)
  
  # 2. Parse Decision
  if "Tool_Call" in response:
      tool_name = response.tool
      tool_args = response.args
      
      # 3. Execute Tool
      result = execute(tool_name, tool_args)
      
      # 4. Update Memory
      history.append(result)
  
  else:
      return response.final_answer

Deep Dive: Tools & Function Calling

How does a text model use a calculator? Through Function Calling.

The API Contract

The LLM cannot "click" buttons. Instead, we give it a menu of functions it can "call" by writing specific text.


Example Tools we will build:
  • search_web(query: str)
  • read_pdf(filepath: str)
  • run_python(code: str)
  • send_email(to: str, body: str)

Common Pitfall

Hallucinated Arguments:
Sometimes an agent tries to call a tool that doesn't exist, or invents arguments.

Example: Calling search_web(query="AI", date="tomorrow") when the tool doesn't support a date filter.

Solution: Strict Schema Definition (JSON Schema) is required to constrain the agent.

Deep Dive: Memory & Context

Agents need memory to maintain continuity over days or weeks.

Short-Term

The Context Window

This is the "Working Memory." It contains the current chat history. It is fast but limited (e.g., 8k - 128k tokens).

Long-Term (RAG)

Vector Database

We convert documents into numbers (embeddings) and store them. The agent "searches" this DB to recall facts from months ago.

Stateful

Checkpointing

If an agent crashes, it needs to resume exactly where it left off. We save the "State" (current step, variables) to a database.

Real-World Case Study

The "Thesis Validator" Agent. This is Use Case #3 in our syllabus.

The Challenge

A professor needs to check if a student's thesis topic is actually novel or if it has been done before.

The Agent Workflow

Perception: Agent reads the PDF uploaded by the student.
Extraction: Agent uses an LLM to extract the "Core Hypothesis" (e.g., "AI improves farming yield by 20%").
Investigation (Loop):
- Tool: Search Google Scholar for "AI farming yield".
- Tool: Download top 3 related PDFs.
- Tool: Compare Student Hypothesis vs. Existing Papers.
Reporting: Agent generates a "Novelty Score" and writes a report citing the overlapping papers.

Orchestration Patterns

How do we manage multiple agents working together?

Supervisor-Worker (The Boss)

A central "Supervisor Agent" (usually GPT-4) breaks down the plan and delegates tasks to "Worker Agents" (smaller, specialized models).

Example: Supervisor tells "Researcher" to find data, then tells "Writer" to format it.

Sequential Handoffs (The Relay)

Agent A finishes a task and passes the entire context to Agent B. No central boss.

Example: "Email Agent" receives a complaint -> passes to "Support Agent" -> passes to "Ticket Agent".
